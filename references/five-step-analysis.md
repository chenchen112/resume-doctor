# 五步分析流程详细指南

## 概述

五步分析是 resume-doctor 的核心分析方法，通过系统化的评估流程，为用户提供全面的简历优化建议。

## 第一步：整体评分 (10分制)

### 评分框架

#### 执行方法
1. **信息完整性检查**
   - 工作经历完整性（公司、职位、时间）
   - 项目经验详细程度（至少1个完整项目）
   - 技术栈覆盖范围（核心技能是否明确）
   - 教育背景和证书信息

2. **数据量化程度评估**
   - 业务指标量化（用户数、收入、效率提升）
   - 技术指标量化（性能优化、系统容量）
   - 时间维度量化（项目周期、维护时长）

3. **减分项识别与分类**
   - 高危减分项（严重影响评分）
   - 一般减分项（中等影响）
   - 轻微问题（可优化项）

4. **综合评分计算**
   - 基础分：信息完整性（40%）
   - 质量分：数据量化程度（40%）
   - 减分项：问题严重程度（20%）

### 评分标准详解

| 分数区间 | 等级 | 特征描述 | 典型表现 |
|---------|------|----------|----------|
| 9-10分 | 优秀 | 信息完整、数据量化充分、几乎无减分项 | "主导电商平台重构，DAU从10万提升至50万" |
| 7-8分 | 良好 | 质量较高，少量可改进项 | "负责核心模块开发，优化响应时间30%" |
| 5-6分 | 一般 | 明显问题，需要较大改进 | "参与系统开发，使用Java技术栈" |
| 3-4分 | 较差 | 多个高危减分项 | "负责开发工作，技术栈：编程语言" |
| 1-2分 | 极差 | 信息严重不足或质量问题严重 | 仅提供岗位名称，无具体内容 |

### 评分原则
- **质量优先**：量化数据和具体成果比内容数量更重要
- **真实性验证**：仅基于用户提供的信息进行评分
- **一致性标准**：相同质量水平的简历获得相同评分

## 第二步：亮点识别

### 亮点特征库（按岗位分类）

#### 前端开发
| 亮点类型 | 具体表现 | 评分权重 |
|----------|----------|----------|
| 性能优化 | 首屏加载时间优化、包大小减少 | 高 |
| 复杂交互 | 动画效果实现、用户体验提升 | 中 |
| 架构设计 | 组件化建设、工程化实践 | 高 |
| 跨平台能力 | 多端适配、响应式设计 | 中 |

#### 后端开发
| 亮点类型 | 具体表现 | 评分权重 |
|----------|----------|----------|
| 高并发处理 | QPS提升、系统容量扩展 | 高 |
| 分布式架构 | 微服务设计、集群部署 | 高 |
| 性能优化 | 响应时间优化、资源利用率提升 | 中 |
| 业务规模 | 大型系统维护、复杂业务逻辑 | 中 |

#### 算法工程
| 亮点类型 | 具体表现 | 评分权重 |
|----------|----------|----------|
| 模型效果 | 准确率/召回率提升 | 高 |
| 数据规模 | 大数据处理、分布式训练 | 高 |
| 业务落地 | 实际业务场景应用 | 中 |
| 学术成果 | 论文发表、专利申请 | 中 |

### 亮点识别流程
1. **关键词扫描**：识别技术术语和量化指标
2. **上下文分析**：评估技术实现的复杂度和影响范围
3. **重要性排序**：按权重和影响力对亮点进行排序
4. **具体化描述**：将抽象亮点转化为具体的技术成果

## 第三步：减分项识别

### 减分项分类体系

#### 高危减分项（严重影响评分）
| 问题类型 | 检测模式 | 改进建议 | 严重程度 |
|----------|----------|----------|----------|
| 模糊描述 | "负责/参与"开头 | 改为"主导/独立负责"+量化成果 | 🔴 高危 |
| 无量化数据 | 缺少数字指标 | 添加具体业务/技术指标 | 🔴 高危 |
| 技术栈模糊 | 技术名无版本 | 明确版本号和具体用法 | 🔴 高危 |

#### 一般减分项（中等影响）
| 问题类型 | 检测模式 | 改进建议 | 严重程度 |
|----------|----------|----------|----------|
| 时间线断层 | 工作经历有空档 | 补充学习或项目经历 | 🟡 中等 |
| 技术堆砌 | 技能列表过长 | 保留核心相关技能 | 🟡 中等 |
| 格式不一致 | 排版混乱 | 统一格式和风格 | 🟡 中等 |

### 岗位特定减分项
- **前端**：缺少TypeScript经验、组件库使用经验
- **后端**：缺少数据库设计经验、业务场景描述
- **算法**：缺少模型细节、训练数据规模描述
- **数据**：缺少ETL流程、数据质量管控经验

## 第四步：优化建议生成

### 建议生成模板库

#### 描述优化模板
```
❌ 问题描述：[模糊/笼统的描述]
✅ 优化建议：[具体化描述] + [量化成果] + [技术细节]
```

#### 量化数据补充模板
```
❌ 原始描述：[缺乏数据的描述]
✅ 优化建议：添加[具体指标]，如[业务指标]提升[X]%，[技术指标]优化[Y]%
```

#### 技术栈明确化模板
```
❌ 模糊技术：[笼统的技术描述]
✅ 明确技术：[具体技术栈] + [版本号] + [应用场景]
```

### 建议个性化适配
- **根据岗位调整**：不同岗位的关注点和评价标准不同
- **根据经验级别调整**：初级/中级/高级工程师的建议重点不同
- **根据行业调整**：互联网/金融/制造业的行业特性差异

## 第五步：ATS关键词分析

### ATS系统工作原理
- **关键词匹配**：基于岗位描述的关键词扫描
- **权重计算**：必备词和加分词的不同权重
- **匹配度评估**：计算简历与岗位的匹配程度

### 关键词库管理

#### 必备关键词（权重70%）
| 岗位 | 核心关键词 | 替代关键词 |
|------|------------|------------|
| 前端 | React, Vue, TypeScript | Angular, Svelte |
| 后端 | Java, Spring, MySQL | Python, Django, PostgreSQL |
| 算法 | Python, TensorFlow | PyTorch, Scikit-learn |

#### 加分关键词（权重30%）
| 岗位 | 进阶关键词 | 专业领域关键词 |
|------|------------|----------------|
| 前端 | Next.js, GraphQL | PWA, WebAssembly |
| 后端 | Spring Cloud, Kafka | 微服务, 分布式事务 |
| 算法 | BERT, Transformer | 强化学习, 联邦学习 |

### 匹配度计算算法
```
匹配度 = (必备词覆盖率 × 70%) + (加分词覆盖率 × 30%)
```

### 输出格式标准化
```markdown
## ATS关键词分析
- **匹配度**：85%
- **已包含关键词**：React, TypeScript, Node.js
- **建议补充关键词**：Next.js, GraphQL, Webpack
- **优化建议**：在项目描述中增加相关技术关键词
```

## 流程质量控制

### 质量检查点
1. **输入验证**：确保简历信息完整性和格式正确性
2. **分析一致性**：确保五步分析结果逻辑一致
3. **建议可行性**：确保优化建议具体可行
4. **输出标准化**：确保报告格式统一规范

### 异常处理机制
- **信息不足**：提示用户补充关键信息
- **格式错误**：提供格式转换建议
- **分析困难**：记录问题并寻求人工协助
